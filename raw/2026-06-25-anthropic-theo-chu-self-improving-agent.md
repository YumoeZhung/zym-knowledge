# Anthropic Theo Chu 演讲：如何构建自我改进的 Agent

机器之心编辑部

最近，Anthropic 团队研究产品经理 Theodora（Theo）Chu 的一段演讲视频，引起了大家的注意。

Theo 表示，当前越来越多的开发者已经不再只是「听说过 Claude」，而是在日常工作中真正感受到效率提升。有人认为 Claude 让自己效率翻倍，也有人认为提升了 10 倍。更重要的是，Claude 已经开始深入 Anthropic 自身的工程流程，「Anthropic 内部超过 80% 的代码由 Claude 合并。」

这意味着，模型的角色正在发生变化。模型不再只是停留在回答问题阶段，而是在一个可以反馈、验证、修正的环境里持续完成任务。

「Close the Loop（闭合循环），给模型一种验证自身输出结果的方式。」

而在这场分享中，Theo 想要告诉开发者的是，「你应该如何适应这个新世界，又应该如何面向未来构建产品，而不是只为过去构建产品。」

为此，Theo 详细拆解了如何构建能够自我改进的 Agent，「真正的配置，是让 Claude 在循环、计划模式和动态工作流中持续运行。」

## 一年之内，模型失败率被大幅压低

Theo 用编程评估基准 SWE-bench Verified 举了一个例子，它由一系列 GitHub issue 组成，模型需要理解问题、修改代码，并通过测试来证明自己真正解决了任务。

一年前的 Sonnet 3.7 得分仅为 60% 左右，而到了 Opus 4.8，得分已经达到了 88%。这意味着，一年前的模型在这些任务上的失败次数，大约是现在的 3 倍。

模型能力提升，并不只是「多做对几道题」，而是失败率正在快速下降。失败率下降之后，模型才有可能承担更长、更复杂、更接近真实工作的任务。

在最新的 Mythos 和 Fable 系列模型中，该基准测试实际上已经出现接近饱和的迹象。

如果你还在用 12 个月前的任务测试今天的模型，就很容易低估模型真正的能力边界。

## 三个核心能力提升

### 一、先规划，再行动

旧模型的典型做法是，上来就开始写大量代码、调用大量工具，几乎没有充分规划。

以 Opus 4.8 为代表的新模型则表现出了自适应思考（Adaptive thinking）的能力。它们会先在内部深思熟虑具体的规范，在预先规划的过程中及时捕捉错误。

建议：要允许模型先思考。使用自适应思考，让模型自己判断什么时候需要思考、需要思考多久。

### 二、错误恢复和自我纠正

旧模型有一个常见问题是 doom looping：模型失败后再次尝试时，往往又会回到之前同样的解法，并不会真正改变做法。

新模型能够读取反馈、理解失败原因，然后尝试不同路径。

开发者需要重新设计模型所处的环境。环境要能给模型反馈，让模型知道自己哪里做错了。

### 三、模型越来越擅长在更长任务周期上运行

旧模型在长任务中经常会「跟丢主线」（Losing the plots）。现在，模型能够稳定地将注意力维持在 100 万个 Token 甚至更高的级别。

未来更合理的方式，是把更完整的任务交给模型。

## 开发者该如何为未来进行构建？

### 一、主动保持野心，动态刷新评估基准（Evals）

Evals 也会过时。AI 时代，Evals 有点像单元测试。一个好的 Eval，不能只测试今天模型已经会做的事情，还应该包含今天模型尚未完全解决、但未来用户体验真正需要的任务。

Evals 要面向未来设计。

### 二、精简「脚手架」（Shrink the Scaffolding）

所谓「脚手架」，即开发者为了修补旧模型的各种漏洞，在其周围套上的系统提示词、外部工具、代码 Harness，以及各种围绕模型搭建的约束和补丁。

Anthropic 自身的例子：团队曾以为新模型在引用功能上出了 Bug，后来发现是新模型遵循指令的能力大幅提升，导致它极其听话地执行了一行很久以前写在系统提示词里、但现在已过时的引用格式指令。删掉那行过时的提示词，功能便恢复正常。

开发者应当「针对意图」去编写简洁的提示词，明确最终想要的结果，而不是围绕着过去老模型的失败经验去过度包装。

### 三、闭环设计，让模型验证自身的输出结果

- 给模型留出思考与工作的空间：引入自适应思考机制，通过投入度拨盘（Effort Dial）让模型自由上调或下调钻研程度。
- 以受控的方式开放高权限：Anthropic 在 Claude Code 中推出了「自动模式」分类器，在控制欲与自主权之间找平衡。
- 提供自我质检的工具：为 Agent 配备如 Computer Use 这样的自动化验证工具，让智能体能够自己去前端点击、做质检。
